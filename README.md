# CUA_Feishu

**专为飞书（Lark/Feishu）开发的 Computer-Use Agent**

基于 [UI-TARS-desktop](https://github.com/bytedance/UI-TARS-desktop) 构建，通过 Vision-Language Model 驱动自然语言指令，实现对飞书客户端的自动化操控。

[简体中文](./README.md)

## 项目简介

CUA_Feishu 是一个基于 UI-TARS-desktop 的 Computer-Use Agent 应用，专注于飞书场景下的桌面自动化操作。用户可以通过自然语言指令，让 AI Agent 自动完成飞书中的各类任务，如：

- 📩 在飞书中自动发送消息、回复
- 📅 创建日程、添加会议
- 📋 操作飞书文档、表格
- 🔍 搜索飞书中的内容
- ⚙️ 调整飞书设置

## 解决的问题与痛点

飞书是企业内部高频使用的办公软件，存在大量重复性 GUI 操作。现有方案存在明显局限：

- **传统 RPA**：依赖控件 ID 或坐标录制，界面稍有变化即失效，不可泛化
- **纯 VLM 方案**：每次从头推理，速度慢、成本高，坐标估算存在偏差
- **无评估手段**：无法量化 Agent 能力，迭代改进缺乏数据支撑

本项目的核心目标：**让 AI Agent 理解飞书界面、执行自然语言指令，并通过记忆机制让重复任务越跑越快、越跑越稳。**

## AI 的关键作用

AI 在本项目中不是辅助工具，而是执行主体，贯穿三个层次：

1. **感知层**：VLM（UI-TARS）通过截图理解当前界面状态，结合 A11Y 无障碍树获得精确的控件坐标，两路信息融合后做出操作决策
2. **执行层**：Agent 将自然语言指令分解为 click/type/scroll 等原子动作序列，逐步完成任务
3. **学习层**：成功执行后自动提取语义嵌入存入记忆库，下次遇到相似指令直接重放，VLM 再做结果验证

## 完整执行闭环

```
用户输入自然语言指令
  → 语义检索历史记忆（Embedding 余弦相似度）
  → 命中（相似度 ≥ 0.84）
      → VLM 验证当前屏幕是否匹配记忆起点
      → 匹配：直接重放操作步骤 → VLM 截图验证任务完成 → 结束
      → 不匹配：注入记忆摘要，LLM 自适应执行
  → 未命中：VLM 推理执行 → 成功后存入记忆
  → 批量测试框架持续评估整体能力
```

同时支持手动录制作为记忆的另一个来源，用户可以示范一次操作直接生成可重放的记忆。

## 实际价值与效率提升

- **重复任务提速 80%+**：记忆命中时完全跳过 LLM 推理，直接重放操作序列
- **坐标精度提升**：A11Y 树提供精确控件坐标，消除 VLM 纯视觉估算偏差，对小按钮、密集列表场景效果尤为显著
- **能力可量化**：批量测试框架量化 Agent 能力，每次迭代后可对比成功率、步骤数、耗时变化，形成数据驱动的改进闭环

## AI 创新亮点

### 双通道感知融合（视觉 + 结构化语义）

主流 GUI Agent 方案只用截图（纯视觉）。本项目在每轮 LLM 调用前自动注入 A11Y 无障碍树快照，格式为预计算的 `<point>X Y</point>`（1000×1000 坐标空间，与 UI-TARS 模型原生输出格式完全对齐）。System Prompt 明确约束模型直接复制坐标，不重新推导，消除"视觉估算 → 自我纠正 → 再估算"的推理抖动。

### 三阶段记忆执行策略

不是简单的"有记忆就重放"，而是带降级的三阶段策略：

| 阶段 | 机制 | 失败处理 |
|------|------|---------|
| 语义检索 | Embedding 余弦相似度 ≥ 0.84 | 直接走正常 LLM 执行 |
| 起点验证 | VLM 截图 + A11Y token 重叠率判断 | 降级为"参考模式"（注入记忆摘要） |
| 结果验证 | VLM 截图验证任务是否真正完成 | 标记失败，不存入记忆 |

### 运行时 C# 编译突破 A11Y 壁垒

飞书使用字节跳动定制 Electron（`frame.dll`），不暴露 Windows UIA Provider，业界通用的 UIA 方案完全失效。本项目发现 IAccessible/MSAA 接口可用，通过运行时动态编译 C# Walker（按源码哈希缓存，只编译一次），绕过 PowerShell 跨进程 COM 编组崩溃问题，成功获取 446+ 节点的完整控件树。

### Embedding 双轨降级

优先调用远端 `/embeddings` API 获取高质量语义向量；网络不可用时自动降级为本地确定性哈希向量（SHA-256 token 哈希映射到 128 维空间，单位化），保证离线场景下记忆检索仍可用。

## 基于 UI-TARS-desktop

本项目基于 [ByteDance UI-TARS-desktop](https://github.com/bytedance/UI-TARS-desktop) 开发。UI-TARS-desktop 是一个基于 [UI-TARS](https://github.com/bytedance/UI-TARS) 模型的原生 GUI Agent 桌面应用，具备以下能力：

- 🤖 由 Vision-Language Model 驱动的自然语言控制
- 🖥️ 截屏与视觉识别支持
- 🎯 精准的鼠标和键盘控制
- 💻 跨平台支持（Windows / macOS / Browser）
- 🔐 完全本地处理，保障隐私安全

## 功能说明

相较于原始 UI-TARS-desktop，本项目针对飞书使用场景进行了以下适配与扩展：

- **Widget 窗口鼠标穿透**：Widget 窗口通过 `setIgnoreMouseEvents(true, { forward: true })` 全局透传鼠标事件，同时在 CSS 层将 `html/body/#root` 设为 `pointer-events: none`，仅对 `button` 等交互元素恢复 `pointer-events: auto`，实现"控制按钮可点击、背景区域完全穿透"的精细分层控制
- **Overlay 标记窗口优化**：在 `operator.ts` 的 `execute()` 中，每次执行动作前调用 `hideOverlay()` 隐藏预测标记窗口，动作完成后在 `finally` 块中调用 `showOverlay()` 恢复，避免透明覆盖层拦截鼠标点击；Windows 下的 `type` 动作额外通过剪贴板粘贴方式输入，绕过输入法兼容问题
- **无障碍树采集**：`getDom` 服务通过运行时动态编译 C# Walker，调用 Windows IAccessible（MSAA）接口遍历飞书进程的完整控件树（446+ 节点），将节点的控件类型、名称、包围盒、可交互状态等字段序列化为 JSON 返回；采集结果以预计算 `point1000` 坐标格式注入 Agent 上下文，辅助 VLM 精确定位控件
- **飞书窗口自动激活**：Agent 执行前自动将飞书窗口置于前台（`ensureFeishuForeground`），避免因窗口遮挡导致操作失败
- **Agent 记忆系统**：完整的任务记忆能力，支持将成功执行的任务步骤持久化为可复用的记忆片段，下次执行相似任务时自动检索并优先回放
  - 基于向量嵌入（Embedding）的语义相似度检索，自动匹配历史任务
  - 三阶段执行策略：语义检索 → 起点验证 → 重放 → VLM 结果验证，任意阶段失败自动降级
  - 手动录制（recording）：用户可手动录制操作步骤并保存为记忆，供 Agent 后续复用
  - 侧边栏 `NavMemories` 组件：可视化浏览、管理所有已保存的记忆条目，支持查看每步截图证据
  - 任务执行阶段可视化：前端实时展示检索、起点判断、重放、验证等执行阶段状态
- **批量测试框架**：完整的 Agent 能力评估体系
  - 测试用例管理：增删改查、拖拽排序、批量启用/禁用
  - 顺序执行所有启用用例，直接复用 `runAgent()` 确保测试环境与生产环境一致
  - 自动生成测试报告：成功率、平均耗时、平均步骤数，每步附截图和思考过程
  - 支持中途停止（epoch 全局停止信号），当前用例执行完毕后安全退出

## 当前已知问题

1. **过程变化导致误操作**：起点验证通过后，执行过程中若出现弹窗、网络加载等界面变化，重放可能在中途失败或产生误操作，缺乏逐步状态追踪机制
2. **敏感操作缺乏安全校验**：发送消息、提交表单、删除内容等不可逆操作在重放路径下直接执行，没有人工确认环节
3. **记忆质量无过滤**：Agent 执行过程中的错误尝试、重复操作仍会被存入记忆，影响后续重放质量
4. **重放容易被环境变化打断**：弹窗、权限提示、网络加载等突发界面变化会导致重放整体失败，缺乏局部恢复能力
5. **缺乏长期错误记忆**：Agent 犯过的错误（如误点相似按钮）不会被记录，相同错误在不同任务中反复出现
6. **测试用例依赖人工编写**：批量测试的用例需要手动创建，无法根据功能文档自动生成，测试计划也不会根据失败结果自动调整

## 未来展望

### 1. 长期经验记忆

从成功任务和失败任务中同时提取经验，建立独立的错误经验库与成功记忆库并行运作。成功任务提炼最短操作路径，多次成功后自动压缩步骤序列；失败任务记录失败上下文（A11Y 快照 + 操作 + 失败原因），下次执行相似任务时将历史错误经验注入 System Prompt，引导 Agent 主动规避已知陷阱。批量测试失败的记忆自动降低权重，连续失败则标记为"待验证"状态，不再走快速重放路径。

### 2. 敏感操作安全校验机制

在执行层引入操作风险分级。对"发送"、"提交"、"删除"、"确认"等高风险操作，先由 VLM 识别操作意图并评估风险等级，再弹出人工确认对话框，展示即将执行的操作截图和目标控件名称，由用户决定是否继续。支持在 Settings 中配置敏感关键词列表，命中则强制触发双重校验流程。批量测试模式下可单独配置安全策略，遇到高风险操作时自动跳过该用例并记录。

### 3. 文档驱动的测试用例自动生成

接入飞书开放平台文档或产品需求文档，用 LLM 解析功能描述，自动生成结构化测试用例。批量测试完成后，将测试报告送给 LLM 分析，自动生成下一轮测试建议：失败用例拆解为更细粒度的子用例，成功率低的功能模块自动增加覆盖，并维护一份"回归测试集"在每次迭代后自动运行，将测试从静态人工维护升级为数据驱动的自进化评估体系。
